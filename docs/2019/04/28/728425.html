每次版本迭代，都是用户想要的吗？

不管外界阻力多大，我希望大家都能坚持我们做产品经理的初衷：为用户做产品，为体验而迭代。你，能做到吗？

作者：得勿

“这个体验不错，但是要花些时间做，我们下个版本再做”
“不要思考那么多，反正产品要一直迭代，先上功能”
“我们这个是第一版，不要浪费时间在体验上，多做点功能”
“你这个方案只是锦上添花，不要纠结了”


这些话语，大家应该耳熟能详吧…


什么时候开始，“迭代”成了堵住大家追求极致的墙？仿佛“体验”只是一个次要的元素，在迭代进程中，永远只是有多余时间才需要做的事情。


为了大家能直观感受这样做的结果，我举一个三岁小孩都懂的例子：


“卖橘子啦！新鲜的大橘子”
你走过来“诶，这个橘子不太一样哦”拿起来闻了一下，涩涩的“肯定很酸吧～”
“呃～这个还没熟很好，以后一定会很甜的。要不要买几斤？”
“我信你个鬼，你个糟老头子…”
“呐呐呐～给你免费试吃一个，这个橘子吃了对身体很好啊，又解渴，营养还丰富”
你犹豫的试吃了一瓣“诶呀，妈呀～这也太太太好吃吃吃了@¥#%#@¥#”你酸得直跳脚。


请问你会买吗？


（口味特殊者、孕妇、牙齿超棒的杠精，请绕道）


一、请停止幻想


为什么你不能接受一个能解渴有营养，只是不合胃口的橘子，却幻想着用户能接受一个功能满足，但体验糟糕的产品呢？


除非你的产品是用户的刚需，且没有竞品。


可惜，大部分产品就像这个橘子一般，只是众多品种的橘子中不起眼的那一种，你凭什么让用户对你的产品未来（迭代）寄予厚望？


二、我经历的一次“迭代死”


背景：当时那个产品在行业中做得还算不错，有很多年的基础，但是该类产品受到微信比较大的影响，决定做一次大改版，一个月后上线。


产品、设计、开发吭哧吭哧，打了鸡血般的，大家都希望这个产品可以脱胎换骨，一展往日雄风。


整体方案经过大家的努力，都觉得非常满意——不论从视觉上，还是交互体验的细节都有了很大改观。


产品最终也如期上线…


当然，很多用户反馈说改版后效果很舒服，比之前更好用了。


可是，有用户提出充值后（下载免费赠送一定金额，之后充值才能正常使用），余额并未增加，骂我们是骗子……


然后你知道的，评分开始崩塌……然后负面评论如潮水般，连我自己进去看介绍，都会觉得这个产品背后肯定是一个骗子公司。


开发也加急处理了，但是因为没有头绪，几天后才解决问题。


结果如大家所料，这个产品越做越差……


写这篇文章的时候，我还下载下来看了一眼——还存在，仅仅只是存在了。


为什么用户不愿意等一等？


为什么我们付出的努力没人看得见，却被用户揪着一个点不放。


我们也回复了，我们在加紧处理问题，为什么没有人愿意等等我们？


人们不会去等一个他觉得不值得信任的人/物。


准时迭代产品版本，真的是用户需要的吗？


为了绩效，为了准时完成任务，为了老板拍脑袋的一句话，为了所谓的执行力，我们忘了迭代的目的，我们忘了是为谁迭代……甚至连核心体验的充分的测试我们都能因此妥协。


三、为什么需要迭代产品？


我对迭代的理解是，阶段性有计划的完成产品目标和用户期待。但是最终的主体是用户，一切只是为了迭代而迭代，忽略用户最终体验的迭代，都是没有意义的。


1. 迭代的价值


  • 阶段性的完成产品的战略目标（战略的调整）

  • 让产品能更好的适应环境的变化（技术的调整）

  • 功能的调整（功能的调整）

  • 用户问题解决（问题解决）


2. 迭代的目标


  • 以产品目标达成，而不是以日期为目标

  • 以产品体验达标，而不是以功能完成度为目标

  • 为用户，而不是为老板


可是，实际情况很不乐观，像我经历的迭代死也不是什么特殊情况。


四、被扭曲的迭代


仔细想想，真的觉得有些夸张，一些被扭曲的迭代，竟带着童年的阴影和天真的幻想。


在我从业10多年中，我看到过这些扭曲的原因，希望大家并引以为戒：


1. 竞品带来的无限压力


竞品太多，太强大，我们的产品被老板提及之时，总不会忘记把业界最好的产品拿过来比较一番；导致产品经理们一个个焦头烂额，仿佛勾起了童年被别人家小孩对比的痛楚。


我们跟无数家长一样，选择了开补习班，恶补差距，而不是停下来思考我们“孩子”的特点和优势的发挥。


越迭代，越失去竞争力——因为同样的功能，团队力量却相差数十倍，如何竞争？


2. 公司上层不懂


领导不懂产品和体验，以为都很简单——抄一抄，分分钟。


这个更为常见，很多新互联网公司老板是做传统产品的，听说互联网人傻钱多，纷纷跑来这里分一杯羹；而他们的理解中，抄起来最快，最保险。


他想运用他们在传统行业中的智慧，将抄袭的低成本做作为产品的优势来获得竞争力。


可是，互联网产品几乎是同台竞争，不像线下产品一样各自为王，没有特色，根本无法参与竞争。


没有哪个真正是靠抄袭成功的，当然，成功之后可以靠抄袭保持成功；而且这样的老板，给的时间一般都特别少。


“那么容易，给你这么多人你还不够用啊？当初我们三个人，一周时间就把这个产品干出来了，不用再讨论了，一周后给我一个新的版本。”


3. 公司领导希望看到自己产品的功能碾压对手


因为老板们并不关心产品的内在逻辑，一般就喜欢看着竞品比功能，生怕功能点少了会丢失竞争力，甚至将竞品范围不断扩大，以获得强大的功能支撑。


最后，老板感觉自己的公司马上要变成一个帝国，开始在脑中飞快的构想蓝图，画大饼。


而设计和开发看了，感受到的却是奴隶般的生活即将开始。


相信我：老板的笑容会随着时间的流逝，开始逐渐凝固的。


——因为那些成熟产品的大而全，是由无数个小而专的功能组合而成；而我们抄袭而成的只是空中楼阁，毫无理念根基，也不知道未来去向。


4. 看多了竞品，以为用户什么都想要


对，你的竞品有ABCD，他们做得都不错。但是，他们的目标用户交集有多少你是否有了解过呢？


不要以为ABCD的功能，都是你们产品的目标用户所需要的。


久而久之，你会慢慢丧失自我，忘记产品的初衷——因为你心中的用户已经和你们实际的用户越来越远。


做产品设计之时，多看看自己的目标受众的特性，少看看竞品。


竞品的作用，是思考产品战略时用的：我要和谁竞争，我要远离谁，我该如何与他差异化竞争。


别人的方案永远不要作为自己的需求，需求只能源自于目标用户。


5. 对“用户体验”的狭隘理解


用户体验是什么？


每个人都能说出标准答案——因为答案可以在百度中搜到，而它却像一段诅咒一样，听得懂它的声音，却无法理会其中的含义。


用户体验，不仅仅是虚空的感受，也不仅仅是效率的提升；它是与产品价值紧密结合的一种产品特质，这种特质让产品更有吸引力，更亲切…它像橘子的味道和香味一样，是与产品价值有机结合的——我们能够通过这些属性来判断水果是否好吃，是否合自己口味；而且它是具有功能性的，它能有效的吸引目标受众，让受众来达成自己的目标。


一定有人说：“味道好不好？这也算功能！”


恩，虽然逻辑上推论，好像不算功能，但是这个世界不是按逻辑来生活的。


人类可以按逻辑去工作，但一定靠感性来生活；一切只依靠逻辑来生活的人，过得都不怎么美好。


为了说明这一点，我们来看看橘子的颜色和味道有什么功能：


了解大自然的人都知道，水果之所以要很美味，是为了让动物们更好的食用它。可是被食用并不是所谓的无私奉献，而是为了自己的一个目标——让动物帮他们在远方播种（因为植物太过密集，生长会受到限制）。


香味和颜色是传播途径，而味道才是让动物们欲罢不能的核心要素；所以味道的功能是：让目标受众依赖你的产品，长期来使用你的产品。


有人说：“饱肚子”才是核心要素啊。


饱肚子方式多了去了，为什么选择吃这个？你们家楼下就有难吃的小饭馆，你为了饱肚子愿意天天在那吃吗？


别骗我了——开车1个小时，然后排队1小时吃饭的都有，只要好吃；更何况旁边还有很多家吃的，你会无动于衷？


6. 以为迭代就是不停的加功能


大家回顾一下，是不是迭代经常都是这种情况。


本次迭代解决XX个问题，新增XX功能，减少…不存在的，我们只有加加加。


因为删除功能非常难，调整功能比较难；而增加功能很容易，因为增加功能对用户的伤害最低！


真的是这样的吗？


功能的无节制的增加只会大幅降低软件的运行效率，并且让用户深陷复杂的界面和选择之中。


我们的产品应该是简洁的，这样用户才能明确他能用来干什么，才能在之后有需要的时候想起你的产品。


为了满足更多用户的需求，而增加主流用户的理解和操作成本，导致产品越来越难用。


——这是典型的，以降低主流用户的体验来提升所有“用户”体验的做法；最终将导致主流用户的流失，他会去寻找更适合自己的产品，而不是一个不伦不类的产品。


好了，以上只是我过往遇到的一些与迭代有关的事情，相信大家可能也经历过，或经历着。我只希望不管外界阻力多大，我们都还是要坚持我们做产品经理的初衷。


我更希望在这个人人都喊用户至上、体验为王的时代，大家真正重视用户和体验，不要以迭代为借口，放弃我们追寻的脚步，更不要让我们的“迭代”被外力扭曲。


为用户而迭代，为体验而迭代，你能做到吗？
